# USER STORY

| Authentication        | AUTH-01       | New user                              | register and verify account with OTP                                                    | access the platform securely with a verified identity                       | AC1: The system shall provide a registration screen with fields: Email, Password, Confirm Password, Display Name, and Register button.<br>AC1.1: `Email` is mandatory, max 254 characters, RFC-compliant format, and trimmed before validation.<br>AC1.2: `Password` is mandatory, length 6-128, must include at least one letter and one number; unsupported characters are rejected with inline error.<br>AC1.3: `Confirm Password` must exactly match `Password`; mismatch disables submit and shows field-level error.<br>AC2: The system shall provide an OTP workflow after successful field validation.<br>AC2.1: OTP request uses a controlled rate limit; when limit is exceeded, system returns a visible countdown and a clear retry message.<br>AC2.2: OTP input is numeric, fixed length (6 digits), and only accepts digits `0-9`.<br>AC2.3: OTP expiration time is displayed in UI and auto-invalidates when timeout is reached.<br>AC3: The system shall process verification and account creation transactionally.<br>AC3.1: Valid OTP creates/activates account and returns success response with authenticated session token.<br>AC3.2: Invalid or expired OTP returns deterministic error code/message without exposing security-sensitive details.<br>AC4: The system shall provide clear post-action behavior.<br>AC4.1: On success, show confirmation message and redirect user to authenticated landing page.<br>AC4.2: On failure, preserve user inputs where safe, highlight invalid fields, and keep user in verification flow.                                                                                                                                                                                                |
| Profile Management    | PROF-01       | Authenticated user                    | update profile settings (display name, skin, avatar configuration)                      | personalize my identity across all realms                                   | AC1: The system shall display a Profile Settings UI with current values loaded from profile store before edit begins.<br>AC1.1: Loading state must include skeleton/disabled controls until profile payload is available.<br>AC1.2: If profile retrieval fails, show recoverable error with Retry action.<br>AC2: The system shall enforce field-level validation rules.<br>AC2.1: `Display Name` is required, length 2-100, leading/trailing spaces removed, blocked-word and invalid symbol checks applied.<br>AC2.2: `Skin` must be selected from dropdown/source list of supported skin IDs from predefined assets catalog.<br>AC2.3: `Avatar Configuration` parts (hair/top/bottom/accessories) must map only to allowed option sets; unknown keys are rejected.<br>AC3: The system shall show dynamic UI behavior during editing.<br>AC3.1: Save button remains disabled until at least one field is changed and all validations pass.<br>AC3.2: Reset/Revert option restores initial profile snapshot without page reload.<br>AC4: The system shall persist updates reliably.<br>AC4.1: Successful save returns updated profile payload and immediately refreshes in-memory UI state.<br>AC4.2: Failed save shows API error banner and preserves pending changes for correction.<br>AC5: The system shall propagate profile effects consistently.<br>AC5.1: Updated display name and avatar settings appear in play session and related panels after refresh/rejoin.<br>AC5.2: Invalid backend payload must not corrupt existing profile values.                                                                                                                                                                                                   |
| Realm Management      | RM-01         | Realm owner                           | create and manage realms with templates and share settings                              | organize virtual spaces for teams and events                                | AC1: The system shall provide a Realm Creation modal with fields: Realm Name, Template dropdown, Privacy toggle, and Create action.<br>AC1.1: `Realm Name` is mandatory, max 200 characters, rejects control characters and duplicate-name edge cases per owner policy.<br>AC1.2: `Template` dropdown data source shall come from supported templates (`home`, `office`, `blank`) with preview metadata.<br>AC1.3: Default template shall be pre-selected; user can change selection before submission.<br>AC2: The system shall initialize map and metadata based on selected template.<br>AC2.1: Selected template loads corresponding default map structure (spawnpoint, rooms, tilemap schema).<br>AC2.2: If template payload is unavailable, system blocks create and shows actionable error.<br>AC3: The system shall support realm access configuration.<br>AC3.1: Owner can toggle `only_owner` mode from settings UI.<br>AC3.2: Share link generation/regeneration must create a new `shareId` and invalidate prior link immediately.<br>AC4: The system shall enforce workflow validation and responses.<br>AC4.1: Create/Save button is enabled only when mandatory fields are valid.<br>AC4.2: On success, show confirmation and insert realm card into list without full page refresh.<br>AC4.3: On failure, show explicit API error and keep modal state for correction.<br>AC5: The system shall handle destructive operation safely.<br>AC5.1: Delete Realm action requires confirmation dialog with realm name acknowledgment.<br>AC5.2: On successful deletion, realm is removed from list and user receives completion notification.<br>AC5.3: On deletion failure, system restores previous UI state and displays root cause message. |
| Real-time Interaction | RT-01         | User in a joined realm                | move and teleport in realtime with synchronized presence                                | collaborate naturally in a shared spatial environment                       | AC1: The system shall establish realtime socket session after successful realm join.<br>AC1.1: Join handshake must include authenticated token and user identifier; invalid handshake is rejected with a user-facing reason.<br>AC1.2: Successful join returns initial position (saved position if exists, otherwise spawnpoint).<br>AC2: The system shall process movement payload with strict validation.<br>AC2.1: Movement payload accepts numeric `x` and `y` within map bounds; non-numeric values are rejected.<br>AC2.2: Invalid movement events are ignored server-side and not broadcast to peers.<br>AC3: The system shall synchronize state to other users in the same room.<br>AC3.1: On valid movement, server emits `playerMoved` with actor ID and coordinates to eligible peers only.<br>AC3.2: Client applies interpolation for smooth avatar transition without desynchronizing authoritative position.<br>AC4: The system shall support teleport transitions.<br>AC4.1: Teleport payload includes `roomIndex`, `x`, `y` and is validated against map definition.<br>AC4.2: Cross-room teleport emits room leave/join events; same-room teleport emits teleport event only.<br>AC5: The system shall provide resilient error handling and recovery.<br>AC5.1: On socket disconnect, user sees connection status and auto-reconnect attempts.<br>AC5.2: On reconnect success, user session and position are restored consistently; on failure, user receives recoverable retry option.<br>AC6: The system shall preserve position continuity.<br>AC6.1: Last valid position is saved on disconnect and used at next join attempt.                                                                                                       |
| Chat                  | CHAT-01       | User in realm or lobby                | send and receive contextual chat messages                                               | communicate with nearby users and channels effectively                      | AC1: The system shall provide chat UI components for room bubble chat, channel chat, DM chat, and lobby chat where applicable.<br>AC1.1: Input box shall support multiline typing with max length indicators (`300` for in-world bubble, `500` for persistent channel).<br>AC1.2: Send action is disabled for empty/whitespace-only messages.<br>AC2: The system shall validate message content before sending.<br>AC2.1: Message text is trimmed and normalized (extra spaces collapsed).<br>AC2.2: Over-limit messages are blocked and must display clear validation feedback.<br>AC2.3: Unsupported payload types are rejected by schema validation and logged.<br>AC3: The system shall support channel membership logic.<br>AC3.1: Channel/DM dropdown source is loaded from chat channel API scoped by realm and membership.<br>AC3.2: DM channel access is allowed only for member IDs listed in channel metadata.<br>AC4: The system shall persist and deliver messages.<br>AC4.1: Persistent messages store sender ID, sender name, channel ID, timestamp, and content.<br>AC4.2: Realtime delivery event is broadcast only to subscribers of the target channel/lobby.<br>AC5: The system shall handle history and failures clearly.<br>AC5.1: Chat history retrieval supports pagination and stable ordering.<br>AC5.2: If message send fails, UI shows non-blocking error and allows retry without losing draft.<br>AC5.3: On success, sent message appears immediately in sender UI with confirmed state.                                                                                                                                                                                                                                    |
| Calling               | CALL-01       | User within proximity range           | initiate, accept, or reject call requests                                               | start voice/video collaboration instantly                                   | AC1: The system shall calculate and publish proximity state for eligible users.<br>AC1.1: Proximity thresholds are evaluated on movement/teleport events.<br>AC1.2: `proximityId` updates are emitted only to impacted users.<br>AC2: The system shall provide call request workflow UI.<br>AC2.1: Caller can select eligible nearby user and send request.<br>AC2.2: Target receives modal/prompt with caller name, timeout, and actions (`Accept`/`Reject`).<br>AC3: The system shall validate call request payloads and session eligibility.<br>AC3.1: Missing or invalid target user ID returns request failure response.<br>AC3.2: Request to offline/non-session user is rejected with clear message.<br>AC4: The system shall process acceptance and rejection reliably.<br>AC4.1: Accept emits `callAccepted` to both participants with shared context.<br>AC4.2: Reject emits `callRejected` to caller and closes pending prompt.<br>AC5: The system shall control call UI lifecycle and response states.<br>AC5.1: On accepted call, media panel opens and displays connection status.<br>AC5.2: On rejected/timeout call, user sees reason and can retry.<br>AC5.3: On media initialization failure, system falls back gracefully and provides actionable error message.                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| Events                | EVT-01        | Realm owner or authorized member      | create and manage events with schedule details                                          | plan and communicate activities for participants                            | AC1: The system shall provide Event form UI with fields: Title, Description, Start Time, End Time, Realm, Optional Capacity, Optional Location/Mode.<br>AC1.1: `Title` is required, max 200 characters.<br>AC1.2: `Description` max length is 2000 characters.<br>AC1.3: Date/time inputs shall use 24-hour format and timezone-aware rendering.<br>AC2: The system shall validate schedule rules.<br>AC2.1: Start Time is required and must be earlier than End Time.<br>AC2.2: Past-time creation policy is enforced according to configured business rule.<br>AC3: The system shall provide controlled selectors and data sources.<br>AC3.1: Realm dropdown source is fetched from realms accessible to current user.<br>AC3.2: RSVP status dropdown values are fixed (`going`, `maybe`, `not_going`).<br>AC4: The system shall support create/update/delete workflows.<br>AC4.1: Save/Create button activates only when required fields pass validation.<br>AC4.2: On create success, event appears immediately in list/calendar panel.<br>AC4.3: On update/delete success, corresponding event view refreshes without stale data.<br>AC5: The system shall handle error and confirmation messages.<br>AC5.1: Invalid payload shows field-level error + top-level summary.<br>AC5.2: Delete action requires explicit confirmation prompt.<br>AC5.3: API failure returns clear reason and preserves unsaved edits for retry.                                                                                                                                                                                                                                                                                                                           |
| Resource Library      | RES-01        | Realm owner or contributor            | create and curate learning resources                                                    | share references and materials with realm members                           | AC1: The system shall provide Resource form with fields: Title, Type dropdown, URL/Content, Description, Realm selector.<br>AC1.1: `Title` is required, max 200 characters.<br>AC1.2: `Description` is optional, max 2000 characters.<br>AC2: The system shall enforce type-specific validation.<br>AC2.1: `Type` dropdown values come from configured source (for example: `pdf`, `video`, `article`, `link`, `note`).<br>AC2.2: If type requires URL, URL must be syntactically valid and normalized.<br>AC2.3: If type is text note, content length constraints must be enforced.<br>AC3: The system shall support filtering and retrieval behavior.<br>AC3.1: Library list supports search by title/author/keyword.<br>AC3.2: Type filter dropdown is populated from available resource types in dataset.<br>AC4: The system shall provide full workflow feedback.<br>AC4.1: On successful creation, system shows confirmation and inserts resource in current list view.<br>AC4.2: On failure, system shows detailed error and keeps user inputs.<br>AC5: The system shall enforce permission and deletion safety.<br>AC5.1: Only creator/authorized role can delete resource.<br>AC5.2: Deletion requires confirmation and updates list state after success.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| Forum                 | FORUM-01      | Realm member                          | create threads and replies                                                              | discuss topics asynchronously and retain discussion history                 | AC1: The system shall provide forum UI with thread list, thread detail, and reply composer.<br>AC1.1: Thread create form includes `Title` and `Body` fields.<br>AC1.2: Reply form includes body field with preview/validation state.<br>AC2: The system shall apply text validation constraints.<br>AC2.1: `Thread Title` is required, max 300 characters.<br>AC2.2: `Thread Body` and `Reply Body` are required, max 5000 characters.<br>AC2.3: Empty or whitespace-only submissions are rejected.<br>AC3: The system shall enforce realm-scoped data context.<br>AC3.1: Thread list only loads content belonging to selected realm.<br>AC3.2: Pagination controls provide deterministic ordering and page metadata.<br>AC4: The system shall support moderation-safe operations.<br>AC4.1: Edit/delete actions visible only for owner or authorized moderator.<br>AC4.2: Unauthorized requests return clear access-denied response.<br>AC5: The system shall provide workflow confirmations and recovery.<br>AC5.1: On create/reply success, UI updates immediately without manual refresh.<br>AC5.2: On error, draft content is retained and user receives actionable error message.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| Admin                 | ADM-01        | System administrator                  | manage users, realms, and platform-level content from an admin console                  | maintain platform governance, quality, and operational control              | AC1: The system shall restrict admin module to authenticated users with `admin` role only.<br>AC1.1: Non-admin users attempting `/admin` access receive forbidden response and redirect.<br>AC1.2: Expired/invalid token shall terminate access and require re-authentication.<br>AC2: The system shall provide admin dashboard UI with metrics and management tables.<br>AC2.1: Dashboard cards/charts display user count, realm count, events, resources, forum activity.<br>AC2.2: Admin list views support search, sorting, pagination, and filter dropdowns (role/status/type).<br>AC3: The system shall support user administration workflows.<br>AC3.1: Admin can change user role through controlled role dropdown source (`user`, `admin`).<br>AC3.2: Role update requires confirmation and returns updated record in table.<br>AC4: The system shall support moderation actions with safeguards.<br>AC4.1: Delete actions for users/realms/content require explicit confirmation dialog.<br>AC4.2: Success operations show completion message and refresh list state.<br>AC4.3: Failed operations display API reason and rollback optimistic UI change.<br>AC5: The system shall provide operational traceability behavior.<br>AC5.1: Management responses include consistent status and message payload for audit/debug usage.<br>AC5.2: Bulk or high-impact actions must surface warning text before execution.                                                                                                                                                                                                                                                                                                                               |
| System Stability      | SYS-01        | Any platform user and operations team | use a stable system that validates payloads and recovers gracefully from runtime errors | ensure reliable collaboration without data corruption or unexpected crashes | AC1: The system shall enforce validation on all REST and socket payloads before business processing.<br>AC1.1: Invalid body/query/path/socket schema returns structured error with field mapping where applicable.<br>AC1.2: Invalid payloads must never mutate persistent or in-memory session state.<br>AC2: The system shall provide centralized error handling.<br>AC2.1: Unhandled exceptions are caught by global handler and converted to standard error response.<br>AC2.2: Client receives user-safe error message while internal stack detail remains server-side only.<br>AC3: The system shall support graceful realtime degradation and recovery.<br>AC3.1: On socket disconnect, client UI indicates disconnected state and starts controlled reconnect attempts.<br>AC3.2: On reconnect success, session rehydration restores presence/chat continuity when possible.<br>AC4: The system shall enforce input and format constraints consistently.<br>AC4.1: Numeric fields use strict numeric parsing and reject non-numeric symbols.<br>AC4.2: Date/time fields use canonical format and reject invalid ranges.<br>AC5: The system shall provide deterministic success/failure responses.<br>AC5.1: Successful operations return structured confirmation payload and visible success feedback.<br>AC5.2: Failed operations return actionable message, preserve user input state, and provide retry path.                                                                                                                                                                                                                                                                                                                                  |
